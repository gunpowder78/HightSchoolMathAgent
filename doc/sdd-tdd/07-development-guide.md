# 开发规范与实施指南

## 1. 文档目的

本文件约束工程实现顺序、目录建议、开发流程和交付标准，确保后续代码开发遵循 SDD/TDD，而不是直接堆砌功能。

## 3. 模块边界建议

- `ui`
  - 只处理展示与交互。
- `application`
  - 编排业务流程。
- `tutor_engine`
  - 承担题目解析、归因、引导。
- `practice_engine`
  - 承担同构题生成与练习判定。
- `model_adapters`
  - 隔离具体模型或厂商差异。
- `persistence`
  - 统一数据访问。
- `reporting`
  - 输出学习报告。
- `realtime`
  - 后续处理 WebSocket、音视频与双脑路由。

## 4. 开发顺序建议

### 阶段一：底座

1. 建立项目骨架。
2. 接入本地数据库。
3. 完成基础配置管理。

### 阶段二：核心异步辅导链路

1. 学生画像与策略模板。
2. 题目输入与结构化解析。
3. 错因归因。
4. 引导式输出。
5. 错题记录。

### 阶段三：强化闭环

1. 同构题生成。
2. 练习结果写回画像。
3. 家长报告。

### 阶段四：实时演进

1. 实时网关。
2. 关键帧筛选。
3. 双脑路由。
4. 会话接力与摘要压缩。

## 5. SDD 工作流

每个功能实现必须经历：

1. 写需求或更新需求编号。
2. 写或更新设计说明。
3. 写测试计划与测试样例。
4. 编码实现。
5. 跑测试并记录结果。
6. 更新追踪矩阵。

## 6. TDD 工作流

对于核心能力，建议严格遵循：

1. 先写失败的测试。
2. 用最小实现让测试通过。
3. 重构并保持测试通过。

适用模块：

- 错因归因逻辑。
- 会话状态迁移。
- 同构题约束校验。
- 数据持久化与查询。
- 模型输出结构化解析。

## 7. Definition of Ready

某个开发任务开始前，必须满足：

- 已有需求编号。
- 已确认输入输出结构。
- 已确认边界条件。
- 已确定测试方式。
- 已明确是否影响提示词或数据结构。

## 8. Definition of Done

任务完成时，必须满足：

- 功能可运行。
- 对应测试通过。
- 文档已更新。
- 没有新增未解释的关键技术债。
- 若有模型输出变化，评测样例已更新。

## 9. 编码与配置建议

- 使用清晰的层次边界，避免 UI 直接拼接复杂 Prompt。
- 提示词模板单独版本化，不要散落在业务代码里。
- 结构化输出统一用 schema 校验。
- 将模型调用封装成接口，避免业务逻辑依赖供应商 SDK。

## 10. 日志与调试建议

- 日志记录请求 ID、会话 ID、学生 ID、模型耗时。
- 生产日志中避免直接落敏感原文。
- 对失败案例要能回放到：
  - 输入；
  - 结构化状态；
  - 模型输出；
  - 最终展示内容。

## 11. 提示词变更流程

- 任何教学 Prompt 变更都必须：
  - 说明原因；
  - 指明影响的需求编号；
  - 补充或更新评测样例；
  - 通过回归测试后再合入。

## 12. 建议首批开发任务

- `DEV-01` 项目初始化与配置。
- `DEV-02` SQLite 数据访问层。
- `DEV-03` 学生画像与策略模板。
- `DEV-04` 题目输入与解析。
- `DEV-05` 错因归因接口。
- `DEV-06` 引导式输出接口。
- `DEV-07` 错题本与会话存储。
- `DEV-08` 同构题生成。
- `DEV-09` 周报生成。

## 13. 禁止事项

- 不要在没有需求编号的情况下直接加功能。
- 不要把长 Prompt 直接硬编码到 UI 层。
- 不要让模型自由生成无约束 JSON。
- 不要在没有测试的情况下重构会话状态机。
- 不要在 MVP 初期过早引入复杂基础设施。
