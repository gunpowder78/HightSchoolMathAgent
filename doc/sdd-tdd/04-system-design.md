# 系统设计说明书

## 1. 设计目标

系统设计必须同时满足三件事：

- 让 MVP 足够简单，能在本地快速落地。
- 让教学流程足够稳定，能约束大模型不乱答。
- 让架构具备演进空间，能从异步图像辅导平滑升级到实时多模态双脑架构。

## 2. 总体架构

### 2.1 MVP 分层

- `UI 层`
  - 负责图片上传、文本输入、结果展示、错题回看。
- `Application 层`
  - 负责编排一次辅导会话的完整流程。
- `Tutor Engine`
  - 负责题目解析、错因归因、提示词拼装、引导动作生成。
- `Practice Engine`
  - 负责同构题生成与难度控制。
- `Persistence 层`
  - 负责学生画像、错题记录、会话状态、报告数据存储。
- `Model Adapter 层`
  - 负责封装具体模型 API，隔离供应商差异。

### 2.2 实时阶段扩展

- `Realtime Gateway`
  - 负责音视频流接入、关键帧上报、实时会话控制。
- `Live Brain`
  - 低延迟感知与对话层。
- `Reasoning Brain`
  - 高深度数学推理层。
- `Shared State Store`
  - 为双脑提供统一状态与长期记忆。

## 3. 组件职责

### 3.1 UI 层

- 输入：
  - 文字、图片、后续语音/关键帧。
- 输出：
  - 引导问题、视觉化解释、练习题、错题记录、学习报告。
- 约束：
  - 不承担核心业务逻辑。

### 3.2 Application 层

职责：

- 管理会话生命周期。
- 调用题目解析与诊断流程。
- 决定本轮是继续追问、给提示、还是进入脚手架/练习。
- 写入持久化结果。

### 3.3 Tutor Engine

内部建议拆分为：

- `Problem Parser`
  - 判断信息是否充分，抽取题型与已知条件。
- `Bug Classifier`
  - 输出错因类别与证据。
- `Dialogue Policy`
  - 决定下一个干预动作。
- `Pedagogical Renderer`
  - 把推理结果转成适合学生理解的表达。

### 3.4 Practice Engine

- 读取原题结构。
- 提取核心知识骨架。
- 施加难度与知识点约束。
- 生成同构题与答案校验信息。

### 3.5 Persistence 层

- `SQLite`
  - 保存学生画像、题目、会话、错题本、报告快照。
- `File/BLOB 存储`
  - 保存图片原件或缩略图。
- 后续可扩展：
  - `PostgreSQL`
  - `Vector DB`

### 3.6 Model Adapter

统一接口建议：

- `analyze_problem(input)`
- `classify_bug(context)`
- `generate_guidance(context)`
- `generate_isomorphic_problem(context)`
- `summarize_session(context)`

## 4. 关键流程

### 4.1 MVP 单次辅导流程

1. 学生上传题目或粘贴文本。
2. `Problem Parser` 判断信息是否充分。
3. 若信息不足，系统先追问。
4. 若信息充分，`Bug Classifier` 生成错因判断。
5. `Dialogue Policy` 选择单一最优干预动作。
6. `Pedagogical Renderer` 输出面向学生的话术。
7. 系统记录本轮状态与结果。
8. 若学生完成理解，进入同构题或收尾总结。

### 4.2 家长回顾流程

1. 读取近一段时间的错题与会话记录。
2. 聚合高频错因与模块分布。
3. 结合策略模板给出训练建议。
4. 输出简洁报告。

### 4.3 实时双脑流程

1. `Realtime Gateway` 接收音频与关键帧。
2. `Live Brain` 负责低延迟陪伴、情绪承接与轻量判断。
3. 当问题超出轻量能力时，触发工具调用。
4. `Reasoning Brain` 异步分析关键帧与上下文。
5. 结果写回 `Shared State Store`。
6. `Live Brain` 将严密结果口语化输出给学生。

## 5. 关键设计决策

### 5.1 为什么 MVP 优先 Gradio

- 多模态组件更原生。
- 适合单机快速验证。
- 对长耗时推理更友好。

### 5.2 为什么存储优先 SQLite + WAL

- 单文件、部署简单。
- 本地单用户稳定。
- 后续能迁移到更完整的关系型数据库。

### 5.3 为什么必须保留模型适配层

- 当前研究报告已表明不同模型擅长不同任务。
- 未来必然出现单模型到双脑路由的演进。
- 适配层能避免把业务逻辑绑死在某个模型 SDK 上。

## 6. 可靠性与错误处理

系统必须处理以下异常：

- 图片解析失败。
- 模型响应超时。
- 模型输出缺少必要字段。
- 会话状态写入失败。
- 实时链路断开或关键帧缺失。

处理原则：

- 优先保留用户输入。
- 优先回退到澄清式提问，而不是瞎给答案。
- 保留错误日志与状态快照，便于复盘。

## 7. 安全与隐私

- 学生图像、语音和画像数据仅作辅导目的使用。
- 默认最小化存储，不保存不必要的原始流媒体。
- 输出报告时避免泄露与辅导无关的敏感信息。
- 模型调用层与持久化层要有明确边界，避免调试日志泄露用户内容。

## 8. 可观测性

建议记录以下指标：

- 每轮会话耗时。
- 模型调用次数、失败率、重试率。
- 错因分类分布。
- 追问次数与脚手架触发次数。
- 同构题生成通过率。
- 实时阶段的首字延迟、平均延迟、关键帧命中率。

## 9. 技术债提醒

- 若跳过状态对象设计，后续多轮辅导会快速失控。
- 若跳过模型适配层，后续双脑架构改造成本会显著升高。
- 若跳过评测埋点，后续很难判断“会不会教”而不仅是“会不会答”。
