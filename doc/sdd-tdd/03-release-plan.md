# 项目规划与里程碑

## 1. 规划原则

- 先规格后代码：先稳定目标、流程、输出契约，再进入实现。
- 先 MVP 后增强：先打通“能用”的基础闭环，再追求实时、多模态与复杂运营能力。
- 先高风险验证后扩张：优先验证教学效果、错因归因和状态稳定性，而非界面复杂度。
- 先可测后发布：每个里程碑都必须绑定明确测试门禁。

## 2. 阶段划分

### `M0` 文档与规格冻结

目标：

- 完成项目章程、需求、设计、开发规范、测试方案与追踪矩阵。
- 明确 MVP 技术栈、模块边界、数据模型与提示词协议。

交付物：

- 本目录全部文档。

退出条件：

- `P0` 需求均可映射到设计与测试。

### `M1` 本地 MVP

目标：

- 打通单机本地辅导闭环。
- 支持图像上传、文本提问、错因归因、引导式讲解、错题本保存。

建议范围：

- Web UI：Gradio 优先。
- 存储：SQLite + WAL。
- 模型：单模型起步，保留适配层。

交付物：

- 本地可运行原型。
- 至少一组核心流程自动化测试。
- 小样本人工评测记录。

退出条件：

- `P0` 需求通过验收。
- 关键流程可稳定重复演示。

### `M2` 教学强化与评测闭环

目标：

- 加入同构题生成、家长报告、能力画像更新。
- 建立回归测试集与人工抽检流程。

交付物：

- 结构化同构题生成器。
- 学习报告视图。
- 评测脚本与数据集样本。

退出条件：

- `P1` 需求基本满足。
- 同构题难度控制通过抽检。

### `M3` 实时多模态原型

目标：

- 引入音频流、关键帧触发与实时陪伴反馈。
- 验证双脑架构与异步分析回填。

交付物：

- WebSocket 实时链路原型。
- 实时状态摘要与会话接力机制。
- 双脑路由演示。

退出条件：

- 实时模式具备可感知低延迟。
- 深度推理不阻塞整体交互体验。

## 3. 工作分解结构

### 3.1 产品与教研

- 防御型提分策略模板定义。
- 白名单/黑名单题型规则表。
- 视觉隐喻与话术模板沉淀。
- 错因分类规则设计。

### 3.2 工程实现

- 前端交互界面。
- 模型适配与提示词引擎。
- 会话状态与存储层。
- 练习生成与回顾报告。
- 实时链路与模型路由。

### 3.3 质量保障

- 单元测试。
- 集成测试。
- 提示词与评测集验证。
- 人工教学质量抽检。

## 4. 关键依赖

- 稳定的大模型 API 或本地模型替代方案。
- 图像处理与上传能力。
- 本地数据库持久化能力。
- 后续阶段的 WebSocket 与音视频设备接入能力。

## 5. 风险与缓释

| 风险 | 表现 | 影响 | 缓释策略 |
| :---- | :---- | :---- | :---- |
| 模型幻觉 | 自信但错误的讲解或跳步 | 误导学生 | 严格提示词约束、评测集、人工抽检 |
| 多轮漂移 | 会话后期忘记规则或条件 | 讲解不一致 | 会话摘要、状态对象、轮次压缩 |
| 题目难度失控 | 同构题过难或超纲 | 打击学生自信 | 白名单知识点、负面约束、人工抽检 |
| 状态丢失 | 页面重跑或长任务导致上下文断裂 | 辅导中断 | 会话持久化、异步任务、SQLite WAL |
| 实时链路成本过高 | 多模态 Token 快速膨胀 | 无法长期使用 | 关键帧策略、摘要接力、分层模型路由 |
| 实时视觉误判 | 手写内容识别错误 | 错误辅导 | 关键帧择优、静态复核、低风险默认策略 |

## 6. 资源与节奏建议

### 6.1 MVP 先行顺序

1. 画像与策略模板。
2. 题目输入与结构化解析。
3. 错因归因与苏格拉底引导。
4. 错题本与会话状态。
5. 基础报告。

### 6.2 不建议在 MVP 首周做的事情

- 实时音视频。
- 复杂多角色后台。
- 向量数据库与长期记忆全量实现。
- 大而全题库运营系统。

## 7. Definition of Ready

开发项开始前必须满足：

- 已有需求编号。
- 已有交互草图或输入输出定义。
- 已有测试思路。
- 已明确是否属于 MVP 范围。

## 8. Definition of Done

开发项完成时必须满足：

- 代码或配置可运行。
- 对应测试已通过。
- 文档已更新。
- 需求与测试矩阵已可追溯。
- 不违背教学安全红线。
