# 需求规格说明书

## 1. 文档目的

本文件定义项目的正式需求基线，供产品、设计、开发、测试共同使用。除非另有批准，所有实现与验收都应以本文件中的编号需求为准。

## 2. 角色与场景

### 2.1 角色

- `R1 学生`
  - 上传题目、表达卡点、接受引导、完成巩固练习。
- `R2 家长/陪练者`
  - 配置目标、查看错因、执行陪练、监督进度。
- `R3 系统管理员/开发者`
  - 配置模型、提示词、题型策略、评测规则。

### 2.2 核心使用场景

- `S1` 学生拍照上传错题，系统定位卡点并引导修正。
- `S2` 学生做完讲解后，系统生成同构题并控制难度。
- `S3` 家长查看最近一周的错题类型、常见失误与训练建议。
- `S4` 系统在多轮对话中保留上下文，不重复走偏。
- `S5` 后续实时场景中，系统结合语音与草稿纸关键帧进行陪练。

## 3. 功能需求

### 3.1 用户与画像

- `FR-01 学生画像录入`
  - 系统必须支持录入学生基本信息、目标分数区间、主要薄弱模块、可用学习时长。
  - 优先级：`P0`
  - 验收重点：信息可保存、可编辑、可供后续策略读取。

- `FR-02 学习策略初始化`
  - 系统必须支持为学生选择或生成“防御型提分策略模板”。
  - 模板至少覆盖：目标区间、白名单模块、黑名单模块、题型处理策略。
  - 优先级：`P0`

### 3.2 输入与题目解析

- `FR-03 多模态题目输入`
  - MVP 阶段系统必须支持文本输入与图片上传。
  - 后续阶段应扩展为语音与视频关键帧输入。
  - 优先级：`P0`

- `FR-04 题目结构化解析`
  - 系统必须把用户输入转化为结构化问题对象，至少包括：题干、题型、涉及知识点、学生当前步骤、信息完整性判断。
  - 当信息不足时必须优先追问，不得直接瞎猜。
  - 优先级：`P0`

### 3.3 诊断与辅导

- `FR-05 错因归因`
  - 系统必须将学生错误归因到以下至少三类之一：
    - 概念混淆；
    - 逻辑跳步/分类遗漏；
    - 计算失误。
  - 优先级：`P0`

- `FR-06 苏格拉底式引导`
  - 系统必须默认先提出一个引导动作，而不是直接给完整答案。
  - 单轮应只推进一个最关键认知点。
  - 连续多轮失败后才允许降维脚手架介入。
  - 优先级：`P0`

- `FR-07 视觉降维解释`
  - 对向量、解三角形、三角函数、立体几何等重点模块，系统必须支持视觉化或生活化隐喻解释。
  - 优先级：`P0`

- `FR-08 白名单/黑名单题型约束`
  - 系统必须根据学生策略过滤训练题型。
  - 对明确列入黑名单的高风险题型应提示“超出当前防御范围”，而非强行展开复杂解法。
  - 优先级：`P0`

### 3.4 练习与巩固

- `FR-09 同构题生成`
  - 系统必须生成与原题核心结构等价、但表层叙述或参数不同的巩固题。
  - 同构题默认控制在“高考 90 分保底”难度带。
  - 优先级：`P1`

- `FR-10 难度边界控制`
  - 同构题生成必须支持知识点白名单、负面约束、推理步数上限、数值友好度限制。
  - 优先级：`P1`

- `FR-11 练习反馈闭环`
  - 系统必须记录学生在同构题上的结果，并把结果回写到能力画像。
  - 优先级：`P1`

### 3.5 记录与回顾

- `FR-12 错题本`
  - 系统必须按题目、模块、错因、时间维度保存错题记录。
  - 优先级：`P0`

- `FR-13 会话状态管理`
  - 系统必须维护当前会话的已知条件、已排除路径、当前推理节点、待追问点。
  - 优先级：`P0`

- `FR-14 家长视角报告`
  - 系统必须输出简洁的学习报告，至少包含：
    - 近期高频错因；
    - 建议复习模块；
    - 黑名单污染提醒；
    - 下一步训练建议。
  - 优先级：`P1`

### 3.6 模型与架构

- `FR-15 模型安全约束`
  - 系统必须对提示词施加边界检查、禁止特值法滥用、限制跳步、要求前置条件验证。
  - 优先级：`P0`

- `FR-16 模型路由与降级`
  - 系统应支持将“实时交互层”和“深度推理层”解耦。
  - MVP 可使用单模型，但设计上必须预留双脑架构。
  - 优先级：`P1`

- `FR-17 实时辅导模式`
  - 后续阶段系统应支持实时音频交互、关键帧触发、工具调用与异步结果回填。
  - 优先级：`P2`

## 4. 非功能需求

- `NFR-01 准确性`
  - 关键教学输出必须优先正确而非华丽，允许延迟少量增加但不允许高置信胡编。

- `NFR-02 可解释性`
  - 所有关键辅导动作都应可追溯到错因、知识点或策略规则。

- `NFR-03 可维护性`
  - 提示词、题型策略、模型适配层、存储层必须模块化。

- `NFR-04 稳定性`
  - 本地 MVP 在单用户场景下应支持稳定会话与持久化，不因页面重跑或长推理轻易丢状态。

- `NFR-05 时延`
  - 异步 MVP 可接受秒级响应。
  - 实时模式目标是亚秒级陪伴反馈与秒级深度分析回填。

- `NFR-06 成本可控`
  - 长会话与多模态输入必须支持摘要压缩、关键帧筛选和路由降级。

- `NFR-07 隐私与安全`
  - 学生图像、语音、历史错题和画像数据必须有明确存储边界与最小化原则。

- `NFR-08 可测试性`
  - 每个核心能力必须具备自动化或人工评测入口。

## 5. 优先级与版本范围

- `P0`：没有则 MVP 不成立。
- `P1`：MVP 后第一阶段强化能力。
- `P2`：实时多模态阶段能力。

### 5.1 MVP 必须覆盖

- `FR-01` 至 `FR-08`
- `FR-12` 至 `FR-15`
- `NFR-01` 至 `NFR-04`
- `NFR-07` 至 `NFR-08`

### 5.2 二阶段强化覆盖

- `FR-09` 至 `FR-11`
- `FR-14`
- `FR-16`
- `NFR-05`
- `NFR-06`

### 5.3 三阶段覆盖

- `FR-17`

## 6. 验收总则

- 任何 `P0` 需求未满足，不得宣称 MVP 完成。
- 任何教学输出若违反“信息不足先澄清、默认不直接给完整答案、黑名单题型不强推”三条红线，视为验收失败。
- 任何新增功能若无法映射到需求编号、设计说明和测试用例，视为不合格变更。
