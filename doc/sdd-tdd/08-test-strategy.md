# 测试策略与验收方案

## 1. 测试目标

测试不只验证“能不能运行”，还要验证：

- 需求是否真正被满足；
- 教学行为是否符合预期；
- 模型是否在关键场景下稳定受控；
- 状态、数据与多轮对话是否可靠。

## 2. 测试层次

### 2.1 单元测试

覆盖对象：

- 题目结构化解析。
- 错因归因规则。
- 会话状态迁移。
- 同构题约束校验。
- 报告聚合逻辑。

目标：

- 快速验证纯逻辑与数据处理正确性。

### 2.2 集成测试

覆盖对象：

- 应用层与数据库集成。
- 模型适配层与业务层集成。
- 图片输入到辅导输出的主链路。

目标：

- 验证模块边界是否正确衔接。

### 2.3 端到端测试

覆盖对象：

- 学生上传错题并完成一次辅导。
- 生成同构题并写回记录。
- 家长查看报告。

目标：

- 验证真实用户路径是否可用。

### 2.4 评测集测试

覆盖对象：

- 典型题型与错因场景。
- 多轮辅导表现。
- 黑名单题型拦截。
- 难度控制表现。

目标：

- 验证“会不会教”，不是只看“会不会答”。

### 2.5 人工抽检

覆盖对象：

- 解释是否有打击感。
- 视觉隐喻是否自然。
- 同构题是否真的同构。
- 家长报告是否可执行。

目标：

- 补足自动化难以覆盖的教学质量维度。

## 3. 测试金字塔建议

- 单元测试最多。
- 集成测试覆盖关键链路。
- 端到端测试覆盖核心旅程。
- 评测集与人工抽检聚焦教学质量。

## 4. 需求到测试映射原则

- 每个 `P0` 需求至少有一条自动化验证。
- 每个教学类需求至少有一条人工抽检标准。
- 每个提示词变更至少触发一组回归评测。

## 5. 核心测试主题

### 5.1 输入充分性判断

验证点：

- 题目信息不足时，系统是否先追问。
- 系统是否拒绝脑补缺失条件。

### 5.2 错因归因准确性

验证点：

- 是否能区分概念混淆、逻辑跳步、计算失误。
- 证据描述是否与输入一致。

### 5.3 教学策略符合性

验证点：

- 是否默认采用引导问题而非完整代答。
- 是否一次只推进一个认知点。
- 是否在连续受阻后才进入脚手架模式。

### 5.4 视觉降维表达

验证点：

- 向量、三角函数、立体几何是否优先使用视觉化解释。
- 隐喻是否贴合研究报告设定。

### 5.5 白名单/黑名单保护

验证点：

- 黑名单题型是否被明确识别与限制。
- 是否避免输出高风险误导式解法。

### 5.6 同构题难度控制

验证点：

- 是否保持核心结构等价。
- 是否未引入额外知识点。
- 是否控制在目标难度区间。

### 5.7 多轮状态稳定性

验证点：

- 连续对话后是否保留已知条件与当前目标。
- 摘要压缩后是否仍可继续正确辅导。

## 6. 样例测试用例

| 用例 ID | 覆盖需求 | 场景 | 期望结果 |
| :---- | :---- | :---- | :---- |
| `TC-01` | `FR-04` | 上传信息残缺的题目截图 | 系统先追问，不直接解题 |
| `TC-02` | `FR-05` | 学生把独立事件当互斥事件 | 判定为概念混淆 |
| `TC-03` | `FR-06` | 学生写出中间式但缺公式依据 | 系统追问依据，不直接给后续步骤 |
| `TC-04` | `FR-07` | 向量数量积讲解 | 使用投影或影子类比 |
| `TC-05` | `FR-08` | 输入折叠空间想象黑名单题 | 提示超出当前防御范围 |
| `TC-06` | `FR-09` `FR-10` | 生成同构题 | 结构等价、难度受控 |
| `TC-07` | `FR-12` | 完成一次辅导 | 错题记录成功保存 |
| `TC-08` | `FR-13` | 多轮对话后继续追问 | 系统保留当前卡点与已排除路径 |
| `TC-09` | `FR-14` | 生成周报 | 输出高频错因与下一步建议 |
| `TC-10` | `FR-15` | 模型遇到不完整条件 | 不滥用特值法，不强行继续 |

## 7. 数据集建议

建议建立以下测试集：

- `eval/basic_whitelist.json`
  - 基础白名单题型。
- `eval/blacklist_guard.json`
  - 黑名单题型防护。
- `eval/bug_classification.json`
  - 错因分类样例。
- `eval/socratic_dialogues.json`
  - 多轮引导样例。
- `eval/isomorphic_generation.json`
  - 同构题质量与难度样例。

## 8. 发布门禁

### MVP 发布前必须满足

- `P0` 自动化测试全部通过。
- 至少完成一轮人工教学质量抽检。
- 错题本、会话状态、画像读写稳定。

### 二阶段发布前必须满足

- 同构题评测集通过率达到预设门槛。
- 家长报告经人工评审可执行。

### 实时阶段发布前必须满足

- 关键帧与双脑路由链路稳定。
- 实时交互下不会因阻塞导致长时间静默。

## 9. 回归策略

- 变更 Prompt：跑评测集 + 抽检。
- 变更状态结构：跑单元 + 集成 + 历史会话恢复测试。
- 变更模型适配层：跑主链路端到端测试。
- 变更同构题逻辑：跑生成评测集与人工难度抽检。

## 10. 缺陷分级建议

- `S1`
  - 直接误导学生、输出错误完整答案、泄露隐私。
- `S2`
  - 错因误判、难度失控、状态严重漂移。
- `S3`
  - 报告不准确、表现不稳定但可恢复。
- `S4`
  - 文案或展示瑕疵。
