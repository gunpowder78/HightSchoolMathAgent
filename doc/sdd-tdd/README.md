# HightSchoolMathAgent Documentation Set

本目录沉淀项目的 `SDD`（Specification-Driven Development，规格驱动开发）与 `TDD`（Test-Driven Development，测试驱动开发）文档基线。

当前仓库尚处于“研究报告已完成、产品与工程文档待落地”的阶段，因此本套文档以“先规格、后实现、再验证”为原则，服务后续完整开发。

## 文档目标

- 把研究报告中的教育学、产品、模型与架构结论转化为可执行规格。
- 为本地 MVP、评测闭环、实时多模态演进提供统一工程语言。
- 为后续代码开发建立清晰的需求编号、设计边界、测试入口和验收标准。

## 建议使用顺序

1. 先读 `01` 与 `02`，统一项目目标与范围。
2. 再读 `03` 与 `04`，确认迭代顺序与架构约束。
3. 开发教学能力前阅读 `05`，实现状态与数据前阅读 `06`。
4. 开发实施遵循 `07`，测试与验收严格参照 `08`。
5. 若出现需求变更或实现分歧，以 `09` 为追踪与回溯入口。

## SDD/TDD 工作约定

- 任何新功能先补充或修改需求编号，再进入设计与开发。
- 任何核心模块先写测试计划与验收标准，再开始编码。
- 任何提示词、工作流、模型路由规则都视为“可测试设计资产”，需要版本化。
- 任何发布都必须同时满足：
  - 需求可追踪；
  - 关键路径有自动化验证；
  - 教学安全与幻觉防护有人工抽检记录。

## 当前阶段结论

- 本项目的首发形态应为“本地单机、图像上传 + 文本/语音引导”的 MVP。
- 教学策略必须以“视觉降维、苏格拉底式引导、防御型提分”为核心。
- 模型架构应预留从单模型到“双脑架构”的平滑演进路径。
- 数据与状态层从一开始就要为错题本、会话记忆、评测闭环留接口。
