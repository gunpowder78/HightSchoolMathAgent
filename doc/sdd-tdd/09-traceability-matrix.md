# 需求追踪矩阵

## 1. 目的

本矩阵用于把研究报告中的关键结论追踪到需求、设计与测试，确保后续开发不偏离项目初衷。

## 2. 追踪矩阵

| 来源结论 | 工程化决策 | 需求编号 | 设计落点 | 测试落点 |
| :---- | :---- | :---- | :---- | :---- |
| 艺术生更适合视觉化、空间化、隐喻化讲解 | 将视觉降维作为默认教学策略 | `FR-07` | `05` 视觉降维策略 | `TC-04` |
| 不能依赖死记模板和固定题号套路 | 构建结构化解析与错因诊断，而不是模板答题 | `FR-04` `FR-05` | `04` Tutor Engine | `TC-01` `TC-02` |
| 防御型提分要依赖白名单/黑名单题型控制 | 系统内置题型策略与拒答边界 | `FR-02` `FR-08` | `02` 策略初始化，`05` 安全红线 | `TC-05` |
| 重点在基础题、中档题与稳态得分 | 同构题默认控制在保底难度区间 | `FR-09` `FR-10` | `05` 同构题生成约束 | `TC-06` |
| 大模型容易算理幻觉、跳步与特值法滥用 | 强制边界检查与结构化输出 | `FR-15` | `05` 系统提示词结构 | `TC-10` |
| 教学应采用苏格拉底式渐进引导 | 系统默认单一干预动作与脚手架流程 | `FR-06` | `05` 苏格拉底工作流 | `TC-03` |
| 需要错题诊断与渐进式修复 | 保存诊断结果并形成错题本 | `FR-05` `FR-12` | `06` ErrorDiagnosis、ProblemRecord | `TC-02` `TC-07` |
| 多轮对话容易漂移 | 建立结构化会话状态与摘要机制 | `FR-13` | `06` 会话状态对象、摘要机制 | `TC-08` |
| 家长需要可执行反馈而非只有题解 | 输出家长视角学习报告 | `FR-14` | `06` WeeklyReport，`07` reporting 模块 | `TC-09` |
| MVP 应先本地单机闭环 | 首发优先 Gradio + SQLite + 单模型适配层 | `NFR-03` `NFR-04` | `03` 里程碑 M1，`04` MVP 分层 | 集成测试、E2E |
| 未来需要实时多模态 | 设计 Realtime Gateway 与关键帧机制 | `FR-17` | `04` 实时阶段扩展，`06` 关键帧记录 | 实时链路测试 |
| 没有单一模型能同时兼顾低延迟与深推理 | 预留双脑架构与模型路由层 | `FR-16` `FR-17` | `04` 实时双脑流程 | 集成测试、实时回填测试 |
| 成本与上下文窗口会成为实时阶段瓶颈 | 设计摘要接力与状态压缩机制 | `NFR-05` `NFR-06` | `06` 会话摘要机制 | 长会话回归测试 |
| 产品必须避免打击感和误导 | 把不直接代答、黑名单保护、脚手架兜底写入红线 | `FR-06` `FR-08` `NFR-01` | `05` 模型安全红线 | 人工抽检 |

## 3. 使用方法

- 任何新增需求都应补充追踪关系。
- 任何需求变更都应同步更新设计落点和测试落点。
- 如果某条研究结论无法映射到需求或测试，说明文档体系仍不完整。
